## 2. 보고서 사양

### 2.1 보고서 유형 및 트랙
- **유형**: 중간 보고서 (진행률 60%, 리뷰 & 계획 필요)
- **권장 트랙**: Standard Flow
- **복잡도 레벨**: ⭐⭐⭐ (3/4) - 기술 분석 필요, 의사결정 지원

### 2.2 권장 분량 및 구조
- **전체 분량**: 4-6페이지
- **구조**: 7란 구조 (중간 보고서 표준)
  1. 요약 (Executive Summary) - 0.5쪽
  2. 배경 및 목표 - 0.5쪽
  3. 진행 현황 (60% 상세) - 1쪽
  4. 기술 성과 (성능 개선 결과) - 1쪽
  5. 발견된 리스크 및 해결 방안 - 1쪽
  6. 다음 계획 (남은 40%) - 0.5쪽
  7. 결론 및 권고사항 - 0.5쪽

### 2.3 상세 수준

#### 기술적 깊이
- **기술팀장**: 높음 (아키텍처 다이어그램, 성능 벤치마크 수치)
- **PM**: 낮음 (기술 용어는 한국어로 쉽게 설명, 비즈니스 임팩트 중심)
- **개발팀원**: 매우 높음 (Redis 캐싱 전략 상세, 쿼리 최적화 예시 코드)

#### 용어 설명 규칙
- **Redis 캐싱**: 기술팀장/개발자는 정의 불필요, PM에게는 "자주 사용하는 데이터를 빠르게 접근할 수 있는 메모리 저장소"로 설명
- **응답 시간 (Latency)**: 모두에게 "사용자가 요청 후 답변을 받기까지의 시간"으로 통일
- **쿼리 최적화**: PM에게는 "데이터베이스에서 원하는 정보를 더 빠르게 찾기"로 단순화

### 2.4 핵심 전달 메시지

#### 1순위 (모든 독자)
- **현재 진행 상황**: 60% 달성, 예정 일정 추적 중 (또는 위험 알림)
- **비즈니스 임팩트**: API 응답 시간 X초 → Y초로 개선 (구체적 수치 필요)

#### 2순위 (기술팀장 + PM)
- **리스크 요소**: 발견된 병목 지점 및 해결 방안
- **의사결정 필요 사항**: 리소스 추가 필요 여부, 마감 연장 필요 여부

#### 3순위 (개발팀원)
- **기술 상세**: Redis 캐싱 설정, 쿼리 최적화 구체 사례
- **다음 작업**: 남은 40% 작업의 우선순위 및 담당자

## 3. 작성 가이드라인

### 3.1 톤과 스타일
- **격식 수준**: 반격식 (존댓글, 자연스러운 문어체)
- **객관성**: 사실 기반 (추측 표현 금지: "~일 것 같다" → "~이다")
- **긍정 톤**: 도전 과제는 "리스크"로 중립적 표현 (부정적 표현 금지)

### 3.2 피해야 할 표현
- "대체로", "어느 정도", "상당히" → 구체적 수치 (예: 30%, 5건)
- "곧", "빠른 시일 내" → 구체적 일자 (예: 2월 15일)
- "~등" → 구체적 나열 (예: "Redis, MySQL, MongoDB")
- "검토한다", "노력한다" → 구체적 행동 (예: "1월 31일까지 성능 테스트 완료")

### 3.3 시각화 요소
- 성능 개선 그래프 (Before/After 벤치마크)
- 진행률 차트 (예: 60% 시각화)
- 리스크 매트릭스 (중요도 × 긴급도)
- Redis/쿼리 최적화 관련 간단한 구조도

### 3.4 Human + AI Readable 구조
- 각 섹션 시작에 한 문장 요약
- 테이블이나 불릿 포인트 활용 (긴 서술 지양)
- 섹션 간 명시적 관계 ("2장에서 설명한 성능 개선으로", "그것" 금지)
- 데이터는 구조화 형식 선호 (Key: Value 또는 테이블)

## 4. 콘텐츠 요구사항

### 4.1 필수 포함 항목
1. **성능 개선 수치** (구체적 측정값)
   - API 응답 시간: X초 → Y초 (감소율 %)
   - 처리량 (Throughput): 변화량
   - 오류율 (Error Rate): 영향도 확인
2. **적용된 기술**
   - Redis 캐싱: 구체적 캐싱 전략 (어떤 데이터를 몇 초 캐싱하는가)
   - 쿼리 최적화: 구체적 사례 (예: Index 추가, JOIN 개선)
3. **진행률 근거** (60%가 정당한지 설명)
   - 완료된 작업: 구체적 목록
   - 남은 작업: 구체적 목록 및 예상 일정
4. **리스크 분석**
   - 발견된 기술 리스크 (최소 2-3건)
   - 각 리스크별 해결 방안
   - 완화 계획 일정

### 4.2 권장 추가 항목
- 성능 테스트 환경 설명 (테스트 대상 API, 부하 수준)
- 기술 채무 (Technical Debt) 영향도 평가
- 다음 단계에서의 병목 예상 지점
- 팀의 학습점 및 베스트 프랙티스

### 4.3 피해야 할 내용
- 너무 상세한 코드 스니펫 (본문에 3줄 이상 금지, 부록에만)
- 미결정된 의사결정 항목 (확정되지 않은 방향 제시 금지)
- 다른 팀에 대한 비판 (객관적 사실만 기술)
- 과거 상황 재론 (현재와 미래에 집중)
