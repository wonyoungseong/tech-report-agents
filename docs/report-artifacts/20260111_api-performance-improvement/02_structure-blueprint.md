## 2. 섹션별 상세 설계

### 섹션 1: 요약 (Executive Summary)

**목적**: 바쁜 독자(기술팀장, PM)가 1분 안에 핵심 파악

**분량**: 0.5쪽 (약 300-400자)

**배열 방식**: 병렬형 (4개 핵심 메시지 독립 제시)

**핵심 내용**:
- 현재 진행률 60% 달성, 예정 일정 대비 상태
- 주요 성과: API 응답 시간 개선율 (구체적 수치)
- 발견된 주요 리스크 2-3건 (한 줄 요약)
- 권고사항: 의사결정 필요 사항 (리소스/일정)

**포함 요소**:
```markdown
| 항목 | 현황 |
|-----|-----|
| 진행률 | 60% (목표 대비 정상/지연) |
| 주요 성과 | API 응답시간 X초 → Y초 (Z% 개선) |
| 주요 리스크 | [리스크 1], [리스크 2] |
| 의사결정 필요 | [예: 추가 인력 2주, 마감 1주 연장 검토] |
```

### 섹션 4: 기술 성과

**목적**: 성능 개선 구체적 수치 제시 (프로젝트 가치 증명)

**배열 방식**: 직렬형 (결과 → 기술 → 분석)

**핵심 내용**:

#### 4.1 성능 개선 결과
- Before/After 비교 (구체적 수치)
- 개선율 계산 및 비즈니스 임팩트

#### 4.2 적용된 기술 상세
- **Redis 캐싱**: 캐싱 전략, 대상 데이터, TTL 설정
- **쿼리 최적화**: 구체적 최적화 사례 (인덱스 추가, JOIN 개선 등)

#### 4.3 벤치마크 분석
- 테스트 환경 설명
- 측정 방법 및 도구

**포함 요소**:
```markdown
## 4.1 성능 개선 결과

| 측정 지표 | Before | After | 개선율 |
|---------|--------|-------|-------|
| API 응답 시간 (평균) | X초 | Y초 | Z% ↓ |
| 처리량 (Throughput) | A req/s | B req/s | C% ↑ |
| P95 응답 시간 | D초 | E초 | F% ↓ |
| 오류율 | G% | H% | I% ↓ |

**비즈니스 임팩트**: 사용자 대기 시간 단축으로 이탈률 예상 감소 (정량 데이터 있으면 추가)

[성능 개선 그래프 - Before/After 비교 차트]

## 4.2 적용된 기술 상세

### Redis 캐싱
- **캐싱 대상**: 상품 정보 조회 API, 사용자 프로필 API
- **TTL 설정**: 상품 정보 60초, 사용자 프로필 300초
- **캐시 히트율**: 85% (목표: 80% 이상)
- **기술적 근거**: 읽기 요청이 쓰기 요청의 10배 이상으로 캐싱 효과 높음

### 쿼리 최적화
- **사례 1**: `products` 테이블에 `category_id` 인덱스 추가 → 조회 시간 500ms → 50ms
- **사례 2**: N+1 쿼리 문제 해결 (Eager Loading 적용) → 10회 쿼리 → 1회 쿼리
- **사례 3**: 불필요한 JOIN 제거 및 필요 컬럼만 SELECT

## 4.3 벤치마크 분석

**테스트 환경**:
- 도구: Apache JMeter
- 부하: 동시 사용자 1000명, 10분간 지속
- 측정 대상 API: `/api/products/list`, `/api/user/profile`

**측정 방법**: 각 API에 대해 100회 요청 후 평균/P95 응답 시간 측정
```

**독자별 주목도**:

- PM: ⭐⭐⭐⭐ (비즈니스 임팩트)
- 개발팀원: ⭐⭐⭐⭐⭐ (기술 상세)

**목적**: 핵심 메시지 재강조 및 의사결정 촉구

**분량**: 0.5쪽 (약 300-400자)

**배열 방식**: 병렬형 (결론 + 권고사항 독립 제시)

**핵심 내용**:

#### 7.1 핵심 결론
- 프로젝트 전반적 평가 (성공 가능성)
- 주요 성과 재언급

#### 7.2 의사결정 필요 사항
- 명확한 의사결정 요청 (누가, 무엇을, 언제)

**포함 요소**:
```markdown
## 7.1 핵심 결론

**현재 상태**: 프로젝트는 60% 진행되었으며, 주요 성과(API 응답 시간 Z% 개선)를 달성하였습니다.

**성공 가능성**: 발견된 리스크에 대한 해결 방안이 수립되었으며, 남은 40% 작업은 계획대로 진행 가능합니다.

**핵심 성과**:
- Redis 캐싱 도입으로 주요 API 응답 시간 30% 개선
- 쿼리 최적화로 데이터베이스 부하 25% 감소
- 시스템 안정성 향상 (오류율 X% → Y%)

## 7.2 의사결정 필요 사항

| 의사결정 항목 | 요청 내용 | 결정권자 | 희망 결정일 |

| 모니터링 도구 라이선스 | 3개월 연장 (비용: XXX원) | 기술팀장 | 2026-01-15 |

**권고사항**:
1. DBA 추가 지원을 승인하여 데이터베이스 마이그레이션 리스크를 완화하시기 바랍니다.
2. 성능 테스트 기간 중 프로덕션 부하 제한을 승인하여 정확한 성능 측정을 보장하시기 바랍니다.
3. 모니터링 도구 라이선스 연장을 승인하여 지속적인 성능 추적을 유지하시기 바랍니다.
```

## 5. Human + AI Readable 구조 적용

### 5.1 자기 완결적 섹션

**원칙**: 각 섹션은 다른 섹션 참조 없이 이해 가능

**적용 예시**:
- 섹션 4에서 "Redis 캐싱"을 언급할 때, 섹션 3의 "완료된 작업"을 참조하지 않고 독립적으로 설명
- 단, 섹션 간 연결이 필요한 경우 명시적으로 "3.2절의 완료된 작업 중..."으로 표기

### 5.2 명시적 관계

**금지 표현**:
- ❌ "위에서 언급한 기술"
- ❌ "그것을 적용하여"
- ❌ "앞서 설명한 대로"

**권장 표현**:
- ✅ "4.2절에서 설명한 Redis 캐싱 기술"
- ✅ "3.2절의 완료된 작업 항목 중 Redis 캐싱 도입"
- ✅ "2.2절의 목표인 API 응답 시간 개선을 위해"

### 5.3 구조화된 데이터

**서술형 대신 테이블/Key-Value 사용**:

**나쁜 예** (서술형):
```
Redis 캐싱을 도입했으며, 상품 정보는 60초, 사용자 프로필은 300초 동안 캐싱합니다.
캐시 히트율은 85%로 목표인 80%를 초과했습니다.
```

**좋은 예** (구조화):
```markdown
| 구분 | 값 |
|-----|-----|
| 캐싱 대상 | 상품 정보, 사용자 프로필 |
| TTL 설정 | 상품 60초, 사용자 300초 |
| 캐시 히트율 | 85% (목표: 80%) |
```

### 5.4 요약 선행

**각 섹션 시작에 요약문 배치**:

**예시**:
```markdown
## 4. 기술 성과

**요약**: Redis 캐싱과 쿼리 최적화를 통해 API 응답 시간을 30% 개선했습니다.
```
